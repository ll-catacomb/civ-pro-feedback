# Change report: civpro-feedback-v3.2.1

Prepared 2026-07-16. No live model calls were made for this recovery.

## Evidence

The v3.2.0 batch was stopped after six saved fixtures all received P from the OpenAI evaluator. That set included both DS answers and the 2019 H answer. Two fixtures completed the dual pipeline; four became single-provider checkpoints after Anthropic returned `overloaded_error` during blind evaluation.

The failure coincided with a bundled prompt change that added DS rarity/quota language and instructed both evaluators and cross-judges to choose the lower adjacent band when uncertain. Those instructions duplicated the already-conservative deterministic 2–2 rule and compressed distinct answers into P. The full four-band raw-answer ladder also enlarged every evaluator context and increased cost.

## Recovery

- Restored the neutral v3.1.1 evaluator and cross-judge prompts.
- Restored the smaller leave-one-exam-out DS/LP anchor pack.
- Kept exact exam points and `whyNotHigher` / `whyNotLower` evaluator fields.
- Kept the deterministic lower hard band only for a genuine four-signal 2–2 split; the hedge and manual-review status remain.
- Kept the evaluator-only sweep, which reuses saved issue maps and sources and writes separately to `.data/evaluator-sweeps.json`.
- Made the sweep staged: choose OpenAI, Claude, or both, and choose a four-fixture smoke set or all eight fixtures. The default is OpenAI over one DS, H, P, and LP fixture (at most four calls), while the complete two-provider sweep remains available at 16 calls.
- Required an explicit provider and scope in the API request and added a browser confirmation that states the exact maximum paid-call count before execution.
- Advanced `PROMPT_VERSION` to `civpro-feedback-v3.2.1` so this recovery state cannot be confused with v3.1.1 or the failed v3.2.0 batch.

## Cost discipline

No additional full batch is required to validate the deterministic tie behavior. Future model-facing experiments should start with the four-call, one-provider smoke screen, add the second provider only if the first result is sane, and use all eight fixtures only for a candidate that survives both smoke screens. A full feedback-chain batch should be reserved for a candidate whose evaluator bands are not compressed.

## Verification

- `npm run check`: passed (lint, TypeScript, 45/45 tests).
- `npm run build`: passed with a clean Next.js production build, including the evaluator-sweep route.
- No model APIs or calibration fixtures were called.
