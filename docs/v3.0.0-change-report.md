# Change report: civpro-feedback-v3.0.0

Prepared for external review (codex). Author: Claude session, 2026-07-16. No calibration batch has been run under v3.0.0 yet; all claims about accuracy below refer to the completed v2.2.0 batch. `npm run check` (41 tests) and `npm run build` pass at the state described here.

## 1. Motivation — what the v2.2.0 audit established

Stage-level accuracy across the eight v2.2.0 fixtures (agreed by two independent audits):

| Stage | Exact | Mean ordinal error |
|---|---:|---:|
| OpenAI provisional evaluator | 4/8 | 0.50 |
| Claude provisional evaluator | 4/8 | 0.50 |
| OpenAI band assessor | 2/8 | 0.875 |
| Claude band assessor | 3/8 | 0.625 |
| Claude cross-judge | 3/8 | 0.625 |
| OpenAI cross-judge | 3/8 | 0.75 |
| Persisted result | 3/8 | 0.75 |

Additional findings this version responds to:

- The separate band assessors regraded the evaluators' defect lists and lost their holistic judgment (OpenAI's assessor turned provisional P/H/H on the 2019 fixtures into H/P/P — every change worse).
- Cross-judges were home-team rubber stamps: the Claude judge matched the Claude chain 8/8, the OpenAI judge matched the OpenAI chain 7/8.
- The confidence tie-break selected OpenAI in 7/8 runs (mean confidence 0.915 vs 0.784) and produced the cohort's only two-band miss (2019-DS: Claude judge H@0.72 lost to OpenAI judge P@0.88).
- No stage across ~30 runs of any version has ever predicted DS.

## 2. Design changes

1. **Banding moved into the blind evaluation.** The evaluator's `provisionalBand` is now the chain's band. The separate `band_calibration` stages (both chains) are deleted. This goes further than the reviewer's earlier recommendation ("give the assessor the provisional band as a nonbinding prior") on the theory that a stage which only ever subtracted accuracy should be removed, not repaired. Contestable — see §8.
2. **Anchor pack** supplied to both evaluations (and to cross-judges when they run). Contents, per exam under review:
   - One instructor-graded **DS** answer and one **LP** answer from a **different exam year** (leave-one-year-out): 2019 and 2014 runs anchor on the 2015 fixtures; 2015 runs anchor on the 2019 fixtures.
   - Up to two same-year **assignment answers** the instructor circulated as best-in-class (`content/course/assignments/<year>-assignment-01-*model-answer*`, first two sorted, frontmatter stripped, truncated at 1,100 words).
   - Framing text: texture calibration only; never doctrinal authority; never infer identities.
3. **Cross-judges run only on provisional disagreement.** They now receive both chains' band positions (`provisionalBand`, `bandRationale`, `strengths`, `priorityGaps`) — previously they saw only the downstream band assessments — plus the anchor pack, and must state explicitly which chain's band they reject and why.
4. **Agreement path replaces the two cross-judges with one light merge call** (`feedback_merge`, Claude judge model, medium effort) that merges the two audited feedback objects into the canonical report without regrading. Falls back to the OpenAI chain's audited feedback on failure.
5. **Confidence-based selection removed entirely.** New deterministic cascade:
   - Step 1: provisional bands agree → persist that band (`decidedBy: "provisional_agreement"`).
   - Step 2: disagree → cross-judges; if they agree → persist (`"judge_agreement"`).
   - Step 3: four signals (two provisionals + two judges) vote; unique majority persists (`"majority"`); a 2-2 split persists the **higher** tied band (`"split"`); both cases are marked manual review with the hedged numeric estimate (LP 1 / P 2 / H 3 / DS 4, mean of the four signals).
   - Judge failures degrade gracefully (`"survivor"` when both fail; the remaining signals still vote).
6. **Evaluation prompt additions:** four-band anchor definitions (previously only in the deleted calibrator prompt), instruction to weigh wrong dispositive bottom lines on heavily weighted questions more severely than localized errors, and anchor-usage discipline.

## 4. Cost and latency estimate (not yet measured)

- Anchor pack: ~15K input tokens × 2 evaluation calls (and × 2 judge calls on disagreement runs). Negligible latency.
- Agreement-path run: ~12 model calls (was ~17); the two band calibrations (~45–130s each) and two cross-judges (~160–215s each) are replaced by one merge (~60–90s). Projected wall time ~9–11 min (was ~13–16), tokens roughly −25%.
- Disagreement-path run: similar to v2.2.0 minus the two band calibrations.

## 5. Blindness analysis

- A run never receives its own fixture's grade: `gradedAnchorFixtures` filters `fixture.examId !== examId` (unit-tested for all three exams).
- A run **does** receive the other year's fixture grades. Consequence recorded in QA.md: exact-band results must be read leave-one-year-out; future graded answers should refresh the anchor pool so benchmark and anchors fully separate. The eight-fixture benchmark is no longer fully independent of the prompt inputs — this is the deliberate trade for band-texture anchoring.
- Assignment exemplars carry no grades and are different assessments; identities are not stripped from the text (the professor circulated them publicly within the course) but prompts forbid referencing or inferring identity. Reviewer may prefer mechanical name-stripping.

## 6. Backward compatibility

- Old runs (v1.x–v2.2.0) still validate (`bandAssessment` optional everywhere, `decidedBy` optional, old `feedbackSource` values still valid) and still render in both UIs; CSV/JSON exports unchanged in shape.
- `hashInput` includes `PROMPT_VERSION`, so input hashes roll over as usual.

## 7. What was recommended but deliberately NOT implemented (open items)

1. **Immutable exam point weights** (reviewer's earlier rec #5 — fabricated normalized weights in issue maps). Not addressed; the rubric prompt still requires weights totaling 100.
2. **Per-stage dashboard metrics / confusion matrix / fractional hedge distance** (rec #6). Not built; audits still require reading `runs.json`.
3. **Dispositive-outcome ledger as schema.** Implemented as prompt language only ("weigh wrong dispositive bottom lines more severely"), not as a structured per-question field.
4. **Repeat-run variance measurement.** Still unmeasured; recommended before attributing the next batch's delta to these changes.

## 8. Specific questions for the reviewer

1. Deleting the band assessor vs. repairing it with the provisional as prior: the delete is justified by measured harm, but it removes the "why not higher / why not lower" boundary articulation that stage produced. Worth re-adding as fields on the evaluation schema?
2. Step-3 tie-break (2-2 split → higher band): defensible given the historical downward skew and zero-DS record, or too opportunistic? Alternative: persist the hedge only and leave `predictedGrade` empty on splits (costs dashboard comparability).
3. The merge call runs on the Claude judge model (better measured post-evaluation accuracy); it could equally alternate or use the cheaper work model since it does not band.
4. Anchor pack composition: DS+LP only (extremes). Should an H or P anchor be added at ~5–6K more tokens per evaluation call, given the P/H boundary is where most misses live?
5. Evaluation prompt now carries banding duty, anchors, and dispositive weighting — it is the longest prompt in the system. Any dilution concern inside the single stage we depend on most?
